我看完这组结果，结论很清楚：

**这台机子跑 Qwen3.5-35B-A3B-FP8 时，核心瓶颈已经是 GPU 算力，不是 CPU，也不是 KV cache 容量本身。** 你改的几个参数，大多只是让排队方式略有变化，**没有把系统天花板真正抬高**。这一点在 steady_rps 和 burst_concurrency 两组里都很明显。 

### 1. 基线表现：已经很接近单卡极限

基线下，steady_rps：

* target 1 时实际约 **1.035**
* target 5 时实际约 **4.684**
* target 20 时实际约 **17 左右**
* target 40 时实际只有 **29.43**
  而这时 GPU compute 平均已经到 **95.87%**，P95 latency 约 **9.11s**，KV cache 平均约 **46.10%**、峰值约 **85.21%**。这说明到了高压区，先顶住你的不是“KV 不够”，而是**算力打满后排队变长**。 

burst_concurrency 也一样：从 concurrency 1 到 20，吞吐涨得很快；再往上加，并不会线性增长，说明并发堆上去后已经进入饱和区。

### 2. `--performance-mode throughput`：几乎没带来本质变化

这个配置在 steady_rps=40 时，实际成功 RPS 约 **29.48**，比基线 **29.43** 只高了一点点；P95 latency 约 **9.10s**，也几乎一样。换句话说，**吞吐模式没有把你的上限明显抬起来**。 

这很符合预期：当瓶颈已经是模型主体算力，而不是调度器开销时，`throughput` 模式提升就不会大。

### 3. `--async-scheduling`：同样基本没实质收益

`--async-scheduling` 在 steady_rps=40 时，实际成功 RPS 约 **29.43**，几乎和基线重合；P95 latency 约 **9.04s**，只是轻微波动。说明你的场景里，**调度不是主瓶颈**。 

一句话：
**不是“不会排”，而是“算不过来”。**

### 4. `--kv-cache-dtype fp8`：收益很小，更多是容量优化，不是算力优化

`--kv-cache-dtype fp8` 在 steady_rps=40 时，实际成功 RPS 约 **29.48**，和基线几乎一样；P95 latency 也仍在 **9.10s** 左右。 

这说明你当前这组任务：

* 输出不算长
* KV cache 占用虽然会上升，但**还没高到把系统彻底卡死**
* 所以把 KV 改成 fp8，主要改善的是**缓存空间效率**
* 但对**实际吞吐上限**帮助不大

也就是说，它不是没用，而是**在你这组 workload 下，不是主矛盾**。 

### 5. `--max-num-seqs 512 --max-num-batched-tokens 8192`：反而是最值得警惕的一组

这组看上去像“加大批处理能力”，但结果并没有更好：

steady_rps=40 时：

* 实际成功 RPS 约 **28.91**
* 比基线 **29.43** 还低一点
* P95 latency 约 **9.52s**
* KV cache 平均约 **55.51%**
* 峰值约 **99.12%**

也就是：**你把可装入的请求数放大了，但没有增加真实算力，于是更多请求只是更深地排队，时延更差，吞吐还没明显涨。**  

更关键的是，这组在 steady_rps=40 时 GPU 显存占用均值还是 76% 左右，但 KV cache 峰值已经接近满格，说明它更容易把系统推到“缓存压力+排队拉长”的区域。

所以这组参数对你现在这个模型和请求形态，不像优化，更像**过度放大排队深度**。

### 6. 为什么我说瓶颈是 GPU compute，不是 CPU / 内存 / 显存

因为几个信号同时成立：

第一，**system CPU usage 全程很低**，大多只有 1%~3% 左右，不像 CPU 侧前后处理或调度打满。 

第二，steady_rps=40 时，多个配置下 **GPU compute avg 都在 94%~97%** 左右，已经非常接近满载。 

第三，改调度类参数几乎不涨吞吐，这正是“主瓶颈不在调度层”的典型现象。 

第四，KV cache 的确会上升，但多数配置在 steady_rps=40 时还没有完全打满；即便换 fp8 KV，吞吐也没显著改善。这说明 **KV 不是第一瓶颈，只是高压下的次级约束**。
