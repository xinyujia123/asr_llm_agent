# Qwen3.5-27B-FP8：exp_0 ~ exp_4 硬件压测简报

## 1. 实验范围

本报告基于仓库目录 `eval_llm/basic_test/hardware_test/Qwen3.5-27B-FP8` 下 `exp_0` 到 `exp_4` 的两类结果文件：

- `hardware_benchmark_burst_concurrency.json`
- `hardware_benchmark_steady_rps.json`

对比目标是：在相同模型 `Qwen/Qwen3.5-27B-FP8`、相同 `gpu-memory-utilization 0.7` 与 `--enable-prefix-caching` 前提下，不同启动参数对高并发能力与稳态吞吐的影响。

## 2. 各实验配置

- **exp_0**：基线配置
- **exp_1**：`--performance-mode throughput`
- **exp_2**：`--kv-cache-dtype fp8`
- **exp_3**：`--async-scheduling`
- **exp_4**：`--max-num-seqs 512 --max-num-batched-tokens 8192`

## 4. 结果解读

### 4.1 总体结论

如果只看“请求吞吐能力”和“高压下延迟控制”，**exp_4 是这五组里最实用的一组**。

原因很直接：

- 在 burst 并发 200 下，**exp_4 的成功 RPS 最高，为 18.09**，高于基线 exp_0 的 17.62。
- 同样在 burst 并发 200 下，**exp_4 的 P95 延迟最低，为 8.286s**，明显好于 exp_0 的 8.805s。
- 在 steady 目标 40 RPS 下，**exp_4 的实际成功 RPS 最高，为 12.998**，也是五组最好。
- exp_4 没有像 exp_1 / exp_3 那样出现明显退化，属于“增益不夸张，但比较稳”的优化。

换句话说，**如果你的目标是更高并发下尽量多吃请求，同时不要把尾延迟拉得更差，exp_4 最值得优先保留**。

### 4.2 exp_1：`--performance-mode throughput` 基本没有带来收益

exp_1 相比基线 exp_0，整体表现非常接近，但多数关键点还略差：

- burst@200 成功 RPS：15.93，低于 exp_0 的 17.62。
- burst 峰值 token 吞吐：40310.79，也低于 exp_0 的 44573.09。
- steady@40 成功 RPS：12.53，低于 exp_0 的 12.71。

这说明在你当前这套 workload 上，**单独打开 `--performance-mode throughput` 并没有形成有效增益**，至少没有转化成更好的请求吞吐。

### 4.3 exp_3：`--async-scheduling` 也没有明显优势

exp_3 的整体走势和基线很接近，但没有稳定超过 exp_0：

- burst@200 成功 RPS 为 16.69，低于 exp_0 的 17.62。
- steady@40 成功 RPS 为 12.58，也略低于 exp_0 的 12.71。
- 延迟层面也没有明显更优。

这说明在你的请求模式下，**异步调度没有成为主要瓶颈突破口**。更可能的瓶颈仍然是 GPU 计算本身，而不是调度层开销。

### 4.4 exp_2：token 吞吐看起来最好，但需要谨慎解释

exp_2 表面上是 token 吞吐最高的一组：

- burst 峰值 token 吞吐：**53926.37**，五组最高。
- steady@40 token 吞吐：**42480.72**，也远高于其他组。

但这里有一个**非常重要的解释前提**：

exp_2 的同批请求里，`total_input_tokens` 明显高于其他实验。例如在 burst 并发 100 下：

- exp_0 / exp_1 / exp_3 / exp_4：`total_input_tokens` 都约为 **246374**
- exp_2：`total_input_tokens` 约为 **324774**

也就是说，**exp_2 的每请求输入 token 数明显更大**。因此：

- 它的 **tokens/s 更高，不等于请求吞吐更高**。
- 从 **成功 RPS** 看，exp_2 并没有胜过 exp_4，甚至 burst@200 仍低于 exp_4。

所以对 exp_2 更合理的结论是：

**`--kv-cache-dtype fp8` 很可能提升了 token 处理效率，但当前这组结果不能直接证明它提升了“同 workload 下”的请求级并发能力。**

如果你后面要严谨下结论，建议把 exp_2 与其他组的输入 token 分布完全对齐后再复测一次。

### 4.5 exp_4：是当前最平衡的一组

exp_4 的特点不是某一项特别夸张，而是**请求级指标最均衡**：

- burst@200：成功 RPS 最高，P95 延迟最好。
- steady@40：实际成功 RPS 最高。
- 没有出现明显的尾部崩坏。

这说明在你当前测试脚本和负载模式下，**`--max-num-seqs 512 --max-num-batched-tokens 8192` 更贴近这块 GPU 的实际可承载区间**，让调度器更好地把 batch 组织起来了。

## 5. 额外观察

### 5.1 五组实验都没有报错，稳定性都不错

所有实验在关键对比点上 `success_rate` 都是 1.0，没有明显错误堆积。这说明这些参数更多是在影响：

- 请求排队方式
- batch 组织效率
- 尾延迟
- token 吞吐

而不是在造成明显不稳定。

### 5.2 真正瓶颈仍然是高压下的计算饱和

无论哪一组，在更高压力下都出现类似现象：

- steady 从 20 RPS 提到 40 RPS 时，**实际成功 RPS 只小幅上升**。
- 平均延迟和 P95 延迟显著拉高。

这说明系统在高压区已经接近饱和，**不是简单加一个调度参数就能把请求吞吐翻倍**。这一点和你之前对“GPU compute 才是主瓶颈”的判断是一致的。

## 6. 最终结论

本轮 `exp_0 ~ exp_4` 对比后，可以给出一个很明确的排序建议：

### 第一推荐：exp_4

如果你现在要选一组更适合上线继续压测的参数，优先建议：

```bash
--max-num-seqs 512 --max-num-batched-tokens 8192
```

原因是它在**请求级吞吐**和**高压延迟控制**上都是当前最优或接近最优，属于最稳妥的提升。

### 第二观察对象：exp_2

`--kv-cache-dtype fp8` 值得继续测，但要先解决一个问题：

- 目前 exp_2 的输入 token 规模与其他实验不一致，导致它的 token 吞吐优势不能直接与其他组做完全公平比较。

因此更建议你把 exp_2 作为“下一轮严格对齐 workload 后再验证”的对象，而不是这轮就直接下最终结论。

### 不建议单独保留：exp_1、exp_3

- `--performance-mode throughput`
- `--async-scheduling`

这两组在这轮结果里都没有体现出足够明显的收益，至少不值得单独作为主配置保留。

## 7. 下一轮测试建议

建议你下一轮直接做两件事：

1. **用 exp_4 作为主线配置继续压测**，再往上试更细粒度的 `max-num-seqs / max-num-batched-tokens` 组合。  
2. **重新复测 exp_2**，确保数据集、输入 token 长度、输出长度分布与 exp_0/1/3/4 完全一致，再看 `fp8 kv cache` 的真实收益。  

这样你就能把“请求级最优配置”和“token 级最优配置”分开看清楚。
